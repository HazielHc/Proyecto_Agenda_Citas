1. Introduccion 
 
1.1 Proposito 
El presente documento describe los requerimientos de software para AgendaBarber, una 
plataforma SaaS orientada a la gestion automatizada de citas para barberias. El sistema integra 
un agente inteligente de WhatsApp basado en Groq API con el modelo Llama 3.1, y una pagina 
web en Next.js, permitiendo que los clientes agenden, confirmen y cancelen citas de forma 
autonoma mientras el negocio gestiona su operacion desde un panel en tiempo real. 
 
1.2 Alcance 
AgendaBarber V2.0 — Stack gratuito cubre: 
• Agente conversacional en WhatsApp mediante Baileys (libreria open source) 
• Agendado de citas desde pagina web en Next.js desplegada en Vercel 
• Motor de tiempos con tolerancia de 10 minutos y recordatorios automaticos 
• Dashboard operativo en tiempo real con Socket.io 
• Gestion de multiples barberos (1 a 5) por negocio 
• Base de datos PostgreSQL via Supabase (gratis hasta 500MB) 
• Cola de jobs con Upstash Redis (gratis hasta 10,000 comandos/dia) 
• Arquitectura multi-tenant: cada barberia opera en su propio espacio de datos

2. Descripcion General del Sistema  
 
2.1 Perspectiva del producto 
AgendaBarber es un sistema independiente multi-tenant que opera como SaaS. Cada barberia 
registrada tiene su propio espacio de datos aislado identificado por id_negocio. El sistema se 
compone de tres interfaces: el agente de WhatsApp (Baileys + Groq), la pagina web del cliente 
(Next.js en Vercel) y el dashboard del negocio (Next.js en Vercel). El backend corre en Node.js 
sobre Oracle Cloud Free y se conecta a Supabase (PostgreSQL) y Upstash Redis.

3. Requerimientos Funcionales  
 
3.1 Modulo de autenticacion y roles 
 
RF-01 Registro de negocio 
El sistema debe permitir que una barberia se registre proporcionando nombre del negocio, 
correo electronico, contrasena y seleccion de plan. La contrasena debe almacenarse con hash 
bcrypt. Al completar el registro se crea automaticamente el perfil de administrador. 
 
RF-02 Inicio de sesion 
El sistema debe autenticar a Administrador y Barbero mediante correo y contrasena. Tras la 
autenticacion exitosa se genera un JWT con duracion de 8 horas. El token debe enviarse en el 
header Authorization de cada peticion al backend. 
 
RF-03 Control de acceso por rol  
El API Gateway debe validar el JWT en cada peticion. El rol Barbero / Recepcionista solo 
puede acceder al dashboard operativo del dia. El rol Administrador tiene acceso total 
incluyendo clientes, reportes y configuracion. 
 
3.2 Modulo de agendado — Canal WhatsApp 
 
RF-04 Conexion via Baileys 
El sistema debe mantener una sesion activa de WhatsApp mediante Baileys. La sesion debe 
persistirse en disco para sobrevivir reinicios del servidor. El sistema debe reconectar 
automaticamente si la sesion se interrumpe. 
 
RF-05 Flujo conversacional del agente  
El agente debe guiar al cliente para agendar una cita en no mas de 5 mensajes siguiendo este 
flujo: saludo y pregunta del servicio, consulta de disponibilidad, presentacion de horarios 
disponibles, solicitud de nombre y apellido, confirmacion del registro. El numero de telefono se 
toma automaticamente del chat de WhatsApp sin pedirlo al cliente. 
 
RF-06 Motor de lenguaje natural con Groq API  
El agente debe usar Groq API con el modelo Llama 3.1 para interpretar mensajes en lenguaje 
natural. El sistema debe construir un prompt con el contexto del negocio (servicios, horarios, 
disponibilidad del dia) y el historial de la conversacion almacenado en Upstash Redis. La 
respuesta de Groq debe incluir el texto para el cliente y la accion estructurada a ejecutar. 

--- PAGE 7 ---
 
RF-07 Persistencia del contexto conversacional  
El sistema debe almacenar el historial de cada conversacion activa en Upstash Redis con una 
clave por numero de telefono. El contexto debe expirar automaticamente tras 30 minutos de 
inactividad para liberar memoria. 
 
3.3 Modulo de agendado — Canal web 
 
RF-08 Calendario de disponibilidad  
La pagina web debe mostrar un calendario visual con los dias disponibles del mes en curso. Al 
seleccionar un dia, debe mostrar los horarios libres calculados en tiempo real desde la base de 
datos, descontando la duracion de cada servicio para evitar traslapes. 
 
RF-09 Formulario de agendado  
El formulario debe solicitar nombre, apellido, numero de telefono, servicio deseado, fecha y 
hora. Al confirmar, el sistema debe registrar la cita en la base de datos y notificar al negocio via 
Socket.io para actualizar el dashboard en tiempo real. 
 
RF-10 Sincronizacion entre canales  
Ambos canales deben compartir la misma base de datos en Supabase. Si un horario se ocupa 
por WhatsApp, la pagina web debe reflejarlo de inmediato. El sistema debe prevenir doble 
reserva mediante una transaccion atomica en PostgreSQL al momento de guardar la cita. 
 
3.4 Cancelacion de citas 
 
RF-11 Cancelacion por WhatsApp  
El agente debe detectar la intencion de cancelar y solicitar confirmacion con la pregunta: Estas 
seguro que deseas cancelar tu cita del [fecha] a las [hora]? Si el cliente responde SI, el sistema 
cancela la cita y libera el horario. Si responde NO, la cita permanece activa. 
 
RF-12 Cancelacion desde pagina web  
La pagina web debe mostrar un modal de confirmacion antes de procesar la cancelacion. El 
modal debe mostrar los detalles de la cita y dos botones: Cancelar cita y Mantener cita. Solo al 
confirmar se ejecuta la cancelacion en la base de datos. 
 
RF-13 Cancelacion por el negocio  

--- PAGE 8 ---
El Administrador o Barbero debe poder cancelar una cita desde el dashboard. El sistema debe 
notificar automaticamente al cliente por WhatsApp indicando que su cita fue cancelada. 
 
3.5 Motor de tiempos 
 
RF-14 Job de tolerancia de 10 minutos  
Al registrar una cita, el sistema debe programar automaticamente un job en Bull Queue con un 
delay igual al tiempo hasta la hora de la cita mas 10 minutos. Cuando el job se ejecuta, si la cita 
no ha cambiado al estado Cliente llego, el sistema debe cambiarla automaticamente a No 
asistio y liberar el horario. 
 
RF-15 Aviso previo al vencimiento  
El sistema debe programar un segundo job que se ejecute 2 minutos antes de cumplirse la 
tolerancia. Este job debe enviar una notificacion push via Socket.io al dashboard del negocio 
indicando el nombre del cliente y los minutos restantes para la cancelacion automatica. 
 
RF-16 Cronometro de servicio  
Cuando el personal confirma la llegada del cliente en el dashboard, el sistema debe registrar la 
hora exacta de llegada en la base de datos e iniciar un cronometro en tiempo real via Socket.io. 
El cronometro debe mostrarse en el dashboard hasta que el servicio sea marcado como 
completado. 
 
RF-17 Calculo de retraso acumulado  
Si el cliente llega despues de su hora agendada pero dentro del periodo de tolerancia, el 
sistema debe calcular el retraso en minutos y notificar automaticamente por WhatsApp al 
siguiente cliente en la agenda indicando la espera aproximada. 
 
RF-18 Liberacion de horario para walk -in 
Cuando una cita es cancelada automaticamente por no asistencia, el sistema debe marcar ese 
horario como Disponible — entrada directa en el dashboard, permitiendo al personal registrar 
manualmente a un cliente walk-in. 
 
RF-19 Lista de espera 
Si un cliente solicita un horario ocupado, el agente debe ofrecer agregarlo a la lista de espera. 
Al liberarse ese horario por cancelacion o no asistencia, el sistema debe notificar 
automaticamente al primer cliente en la lista.

3.7 Modulo de notificaciones automaticas 
 
RF-20 Recordatorio 24 horas antes  
El sistema debe programar un job en Bull Queue al momento de crear cada cita. El job debe 
ejecutarse 24 horas antes de la hora agendada y enviar un mensaje de WhatsApp al cliente 
con los detalles de su cita preguntando si confirma su asistencia. 
 
RF-21 Procesamiento de confirmacion  
Si el cliente responde SI al recordatorio, la cita cambia al estado Confirmada. Si responde NO, 
la cita se cancela automaticamente y el horario queda libre. Cualquier otra respuesta debe ser 
procesada por el agente IA para interpretar la intencion. 
 
RF-22 Mensaje de agradecimiento  
Al marcar una cita como Completada, el sistema debe enviar automaticamente un mensaje de 
WhatsApp al cliente agradeciendo su visita e invitandolo a agendar su proxima cita con un 
enlace directo a la pagina web del negocio. 
 
RF-23 Aviso de espera al siguiente cliente  

--- PAGE 10 ---
Cuando existe retraso acumulado, el sistema debe enviar un mensaje de WhatsApp al 
siguiente cliente en agenda indicando el tiempo estimado de espera antes de su cita. 
 
3.8 Panel de administracion — Dashboard 
 
RF-24 Vista en tiempo real  
El dashboard debe mostrar todas las citas del dia actualizadas en tiempo real via Socket.io sin 
necesidad de recargar la pagina. Cada cita debe mostrar: hora, nombre y apellido del cliente, 
telefono, servicio, barbero asignado, estado actual y cronometro activo cuando el servicio este 
en curso. 
 
RF-25 Acciones rapidas por cita  
El personal debe poder ejecutar las siguientes acciones con un solo clic desde el dashboard: 
confirmar llegada del cliente, iniciar servicio, completar servicio y marcar como no asistio. 
 
RF-26 Horarios libres y walk-in 
El dashboard debe calcular y mostrar dinamicamente los horarios libres del dia basandose en 
las citas activas reales, no en valores estaticos. El personal debe poder registrar un cliente 
walk-in desde un modal que solicita nombre, apellido y servicio. 
 
RF-27 Registro de walk-in 
El sistema debe registrar al cliente walk-in con estado inicial Cliente llego, asignarle el horario 
actual del sistema como hora de cita e iniciar el cronometro de servicio inmediatamente. 
 
3.9 Modulo de gestion de clientes 
 
RF-28 Registro automatico de cliente  
Al agendar una cita por primera vez, el sistema debe registrar automaticamente al cliente en la 
tabla clientes usando el numero de telefono como identificador unico por negocio. Si el cliente 
ya existe, se asocia la nueva cita a su perfil sin volver a pedirle los datos. 
 
RF-29 Historial por cliente 
El Administrador debe poder consultar el historial completo de citas de cada cliente, incluyendo 
fechas, servicios, estados, puntualidad y numero total de inasistencias. 
 
3.10 Modulo de reportes 

--- PAGE 11 ---
 
RF-30 Reporte de citas por periodo  
El sistema debe generar reportes filtrados por dia, semana o mes con datos reales de la base 
de datos, mostrando totales de citas agendadas, completadas, canceladas y no asistidas. 
 
RF-31 Reporte de puntualidad  
El sistema debe calcular el porcentaje de clientes puntuales, tardios dentro de tolerancia y no 
asistidos, usando la diferencia entre hora_agendada y hora_llegada registrada en la base de 
datos. 
 
RF-32 Reporte de servicios populares  
El sistema debe mostrar los servicios mas solicitados en un periodo dado, agrupando por 
id_servicio y contando las citas en estado Completada.

4. Requerimientos No Funcionales  
 
4.1 Rendimiento 
• El agente de WhatsApp debe responder al cliente en menos de 3 segundos 
• Las consultas de disponibilidad deben resolverse en menos de 2 segundos 
• El dashboard debe actualizarse en tiempo real con latencia maxima de 1 segundo via 
Socket.io 
• Groq API con Llama 3.1 ofrece inferencia rapida adecuada para el tiempo de respuesta 
requerido 
 
4.2 Disponibilidad 
• El sistema debe tener disponibilidad minima del 99% mensual 
• La sesion de Baileys debe reconectarse automaticamente ante interrupciones 
• Los jobs de Bull Queue deben persistir en Upstash Redis y reejecutarse si el servidor 
reinicia 
 
4.3 Seguridad 
• Las contrasenas deben almacenarse con hash bcrypt (factor de costo minimo 10) 
• Toda comunicacion entre cliente y servidor debe realizarse sobre HTTPS 
• El JWT debe incluir id_negocio para garantizar aislamiento de datos multi-tenant 
• Cada query a la base de datos debe filtrar obligatoriamente por id_negocio 
• La sesion de Baileys debe almacenarse en disco con permisos restringidos al proceso 
del servidor 
 
4.4 Usabilidad 
• El flujo de agendado por WhatsApp no debe requerir mas de 5 mensajes del cliente 
• El formulario web debe completarse en menos de 2 minutos 
• El dashboard debe ser operable sin capacitacion tecnica por parte del personal 
• Las acciones del dashboard (Llego, Iniciar, Completar) deben ejecutarse con un solo 
clic 
 
4.5 Escalabilidad 
• La arquitectura multi-tenant permite agregar nuevos negocios sin cambios en el codigo 
• Al superar los limites gratuitos (Supabase 500MB, Upstash 10k cmd/dia) la migracion a 
planes pagos debe ser transparente para el sistema 
• Cuando los ingresos del SaaS lo justifiquen, Baileys se migra a Meta Cloud API y Groq 
a Claude API sin rediseno del nucleo 
 

--- PAGE 13 ---
4.6 Compatibilidad 
• La pagina web y el dashboard deben ser compatibles con Chrome, Firefox, Safari y 
Edge modernos 
• No se requiere aplicacion movil en esta version
